# Robustness Findings — 2026-06-02

## Context

This run performed static code analysis of `background.js` to identify failure classes
that prevent the extension from building complex DHIS2 tracker programs (many stages,
multiple `programStageSections` per stage, 8–15 program rules per stage).

Network note: The cloud sandbox has no outbound access to the DHIS2 playground

## Failure Class Fixed: Missing `programStageSections` support

### Error from Step 3 (Simulated Payload Analysis)

**What `createFullProgram` would produce for a complex tracker program:**

```
programStages[i] = {
  id: "...",
  name: "ANC Visit",
  program: { id: "..." },
  sortOrder: 2,
  repeatable: false,
  programStageDataElements: [...],
  // ❌ NO formType — defaults to DEFAULT
  // ❌ NO programStageSections refs
}

// ❌ payload has NO "programStageSections" key at all
```

**What the program rules payload for a HIDESECTION rule would produce:**

```json
{
  "programRuleActionType": "HIDESECTION",
  "programRule": { "id": "..." },
  "programStageSection": { "id": "<model-hallucinated or stale UID>" }
}
```

Because sections are never created, any section UID passed to HIDESECTION either
(a) 409s with "Object not found" if the UID is hallucinated, or (b) silently does
nothing in Capture if the UID was from a prior deleted program.

**Additional bugs in the same code path:**

1. **PRV `valueType` + `useCodeForOptionSet` missing** (`createFullProgram` lines ~11772–11779):
   DE-based `programRuleVariables` were created without `valueType` or
   `useCodeForOptionSet`. This causes option-set comparisons in rule conditions to
   fail silently (the PRV defaults to `TEXT` type without code lookup).

2. **Spurious `programStage` on `DATAELEMENT_NEWEST_EVENT_PROGRAM` PRVs**
   (`createFullProgram` lines ~11777–11778):
   The code set `programRuleVariableSourceType: 'DATAELEMENT_NEWEST_EVENT_PROGRAM'`
   but also added `programStage: { id: stageId }`. Only
   `DATAELEMENT_NEWEST_EVENT_PROGRAM_STAGE` uses a stage reference — combining the
   other sourceType with a programStage reference is invalid and triggers a
   DHIS2 validation warning.

3. **`program_stage_section_id` absent from `create_program` inline rule actions schema**:
   The `program_rules[].actions` schema for `create_program` had no field for
   `program_stage_section_id` or `program_stage_section_name`, making it impossible
   to specify HIDESECTION actions inline even if section IDs were somehow known.

## Fix from Step 4 (DHIS2 API Requirements)

### Correct DHIS2 API payload for a sectioned stage

```json
{
  "programStages": [{
    "id": "stageUid",
    "name": "ANC Visit",
    "program": { "id": "programUid" },
    "formType": "SECTION",
    "programStageSections": [{ "id": "sec1Uid" }, { "id": "sec2Uid" }],
    "programStageDataElements": [...]
  }],
  "programStageSections": [
    {
      "id": "sec1Uid",
      "name": "Vital Signs",
      "programStage": { "id": "stageUid" },
      "dataElements": [{ "id": "de1" }, { "id": "de2" }],
      "sortOrder": 1
    },
    {
      "id": "sec2Uid",
      "name": "Symptoms",
      "programStage": { "id": "stageUid" },
      "dataElements": [{ "id": "de3" }, { "id": "de4" }],
      "sortOrder": 2
    }
  ]
}
```

Key requirements:
- `programStage.formType` must be `"SECTION"` (default `"DEFAULT"` renders sections invisibly)
- `programStage.programStageSections` must list section IDs as back-references
- `programStageSections[i].programStage` must reference the stage UID
- `programStageSections[i].dataElements` is a flat array of `{id}` refs
- In a single `/api/metadata` POST, `programStageSections` must appear **after**
  `programStages` in the payload object so DHIS2's import handler resolves the
  `programStage` back-reference before processing sections

### Correct PRV with `valueType` + `useCodeForOptionSet`

```json
{
  "id": "prvUid",
  "name": "delivery_mode",
  "program": { "id": "programUid" },
  "programRuleVariableSourceType": "DATAELEMENT_NEWEST_EVENT_PROGRAM",
  "valueType": "TEXT",
  "useCodeForOptionSet": true,
  "dataElement": { "id": "deUid" }
}
```

Note: `programStage` must NOT be set when `programRuleVariableSourceType` is
`DATAELEMENT_NEWEST_EVENT_PROGRAM`; it is only valid for
`DATAELEMENT_NEWEST_EVENT_PROGRAM_STAGE`.

- Builds `programStageSections` objects for each stage that has `stage.sections[]`
- Sets `stageObj.formType = 'SECTION'` when sections present

- Retained fallback to `act.program_stage_section_id` for pre-existing sections

**Payload construction**:
- Added `payload.programStageSections = allProgramStageSections` (after `programStages`)

**Summary**:
- Added `stages[i].sections: [{id, name}]` — section IDs per stage
- Added top-level `summary.programStageSections: [{id, name, stageId, stageName}]`

### `background.js` — `addStageToProgram` function

Same section support added: builds `programStageSections`, sets `formType`, returns
section IDs in summary.

### `background.js` — `buildSystemPrompt`

**Internal dependency order**: Added `ProgramStageSections` between `ProgramStages`
and `ProgramRuleVariables`.

**HIDESECTION workflow block**: Added explicit 3-step workflow:
1. Define `sections[]` in `stages[]` at create time
2. Use `program_stage_section_name` in inline rules for auto-resolution
3. Use `manage_program_rules(action=create)` with `program_stage_section_id` from summary
for post-creation rules. Includes ⚠ warning about silent no-op on wrong section UID.

Before this fix, any attempt to build a complex tracker program with sectioned stages
would silently produce stages with `formType: DEFAULT` (sections invisible in Capture),
no `programStageSections` objects on the server, and any HIDESECTION rule actions

## Failure Class Fixed (Round 2): No path to add sections post-creation

### Problem

Even with the Round 1 fix, programs already created without sections had no supported
escape hatch. The only `manage_metadata` actions available were destructive or unrelated
(`delete`, `remove_from_stage`, etc.). A model asked to add a section to an existing stage
had to resort to undocumented `dhis2_query` PUT calls with no validation, no formType
update, and no safety gates.

### Fix: `manage_metadata` action `create_stage_section`

**Schema changes:**
- `action` enum extended with `'create_stage_section'`
- Two new parameters added:
  - `section_name` (string) — display name for the new section
  - `section_data_element_ids` (array of strings) — UIDs of DEs already in the stage

**Handler (`executeManageMetadata`):**

1. Validates `stage_id`, `section_name`, `section_data_element_ids` all present
2. Fetches full stage from DHIS2 (`programStageDataElements` + `programStageSections`)
3. Validates every requested DE UID is actually in the stage (returns helpful error
   with `stage_data_element_ids` if any are missing)
4. Generates a new section UID, computes `sortOrder = max(existing) + 1`
5. Builds atomic `/api/metadata` payload:
   - Updated `programStages` entry with `formType: 'SECTION'` and section ID appended
     to `programStageSections` back-references
   - New `programStageSections` entry with `dataElements`, `programStage`, `sortOrder`,
     and default `renderType`
6. Posts via `postMetadataPayload` (validate → commit)
7. Returns `section_id` in success response so caller can immediately wire a HIDESECTION
   rule using `program_stage_section_id`

**System prompt update:**
- `manage_metadata` description block gains a `create_stage_section` example showing
  the 2-step flow: create section → use returned `section_id` in HIDESECTION rule

### Impact

Before: adding a section to an existing stage required raw `dhis2_query` PUT calls
with no validation, no `formType` promotion, and no UID returned — HIDESECTION rules
still couldn't target the section reliably.

After: `manage_metadata(action=create_stage_section, ...)` is a first-class operation
that validates inputs, promotes `formType` to `SECTION`, and returns the section UID
ready for immediate use in program rule actions.

## Round 3 Addition: Smart PRV sourceType for inline create_program rules

### Error (still present after Round 2)

`createFullProgram`'s inline PRV builder always used `DATAELEMENT_NEWEST_EVENT_PROGRAM`
for every DE reference — even for same-stage HIDEFIELD/SHOWWARNING rules.

`DATAELEMENT_NEWEST_EVENT_PROGRAM` reads from the most recently **saved** event, not the
current form being filled. For a new enrollment's first event, there is no prior saved
event, so the PRV returns null → rule never fires → HIDEFIELD / SHOWWARNING don't trigger.
Even on second visits, the rule reads the **previous** saved value, not what the user is
currently typing.

`_buildAndPostProgramRules` (used by `manage_program_rules(action=create)`) already had
smart `pickSourceType()` logic. The inline builder in `createFullProgram` was inconsistent.

### Fix Applied

Added per-DE source type selection inside the inline PRV loop:
- Iterates `stages[]` to find which stage the DE belongs to
- Builds the set of DE names in that stage
- For each action in the current rule: if the action type is a same-stage type
  (SHOWWARNING, SHOWERROR, WARNINGONCOMPLETE, ERRORONCOMPLETE, SHOWWARNINGINFORMATION,
  SETMANDATORYFIELD, DISPLAYTEXT) OR the action's `data_element_name` is in the same
  stage as the condition DE → `DATAELEMENT_CURRENT_EVENT`
- Otherwise → `DATAELEMENT_NEWEST_EVENT_PROGRAM` (cross-stage read)

`DATAELEMENT_CURRENT_EVENT` evaluates the form's live in-memory value, so
HIDEFIELD/SHOWWARNING fire as the user types. `DATAELEMENT_NEWEST_EVENT_PROGRAM`
is preserved for cross-stage lookups (e.g. "hide Delivery field if Lab Result was X").

### Impact

Same-stage HIDEFIELD, SHOWWARNING, SHOWERROR, SETMANDATORYFIELD rules now evaluate
in real-time during data entry when defined inline in `create_program`. Previously they
appeared to "not work" until the user saved the event and reopened it.
